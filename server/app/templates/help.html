🔍 Health Status & Error Legend

📅 Meteorological Day Boundary

The data day runs from 8:45 AM (prev day) → 8:30 AM (today) — 96 slots of 15 minutes each.
• YDY Sent = how many of yesterday's 96 records the server actually received via all channels.
• YDY Stored = how many records the device successfully saved to its internal memory.
• A station is fully healthy if ≥86 YDY records are on server and the 8:30 AM closing record arrived.

Flag Definitions

Flag Stands For Trigger Rule What It Means
PD Partial Data 96 − YDY_sent ≥ 10 ≥10 records from yesterday are missing from server. Indicates intermittent GPRS or local memory issues.
Suffix (X/96): Shows "Arrival Efficiency" — total records on server vs 96 expected.
NDM Night Data Missing ndm_cnt > 18 (of 36 night slots) >50% of night slots (9 PM → 6 AM) were not transmitted live. Backlog recovery later does NOT clear this. Primary indicator of battery/solar failure at night.
CDM Closing Data Missing cdm_sts = FAIL The 8:30 AM closing record for yesterday is missing. Ensures the very last packet of the data day arrived.
GPRS-Er GPRS Error sent=0, stored>0 Device recorded data locally but zero records reached the server. Total modem or SIM failure.

Sensor / Hardware Flags (Detection Method)

These codes are reported directly by the ESP32 firmware during its startup and hourly self-diagnostics. They indicate physical sensor or wiring failures.

Code Sensor Detection Method ("How it's Detected")
TH-Er Temp / Humidity
  • I²C Failure: Bus busy or address not responding (Temp_E).
  • Stuck Sensor: Reading hasn't changed by 0.01 for 10 hours (TEMP_STUCK).
  • Unreal Value: Reading outside physical limits of 2°C–45°C (TEMP_UNREAL).
RF-Er Rainfall
  • Spike: >10mm jump in a single 15-min slot (RAIN_SPIKE).
  • Counter Reset: Cumulative count decreased without a rollover (RAIN_RESET).
  • Calculation Error: Internal pulse-to-mm math mismatch (RAIN_CALC).
WS-Er Wind Speed
  • Stuck Sensor: Speed constant at 0.00 while other factors imply wind (WS_STUCK).
  • Unreal Value: Speed recorded > 60 m/s (216 km/h) — likely electrical noise (WS_UNREAL).
WD-Er Wind Direction Detected via ADC Voltage Monitoring. If the potentiometer wiper returns a voltage outside the 0V–3.3V range (Short or Open circuit).
RTC-Er Real-Time Clock Detected if the DS1307 module returns an invalid date/time (e.g. Year 1970) or fails to respond to I²C polling.
MUTEX-Er Modem Contention Indicates the firmware timed out (15s) waiting for the Modem Mutex. Signals internal software contention or a task hanging while holding the modem.
SYS-Er System/Memory General firmware error, such as SPIFFS Full, SD Card Init Failure, or Internal Memory corruption check failure.

Common Verdicts (Dashboard Examples)

Verdict Meaning / Trigger Scenario
✓ OK Fully healthy. All data received, no sensor errors.
✗ PD(X/96) CDM Missing records + 8:30 missing. Normal for New Installations on Day 1.
✗ NDM(X/36) Data is complete, but night transmissions failed. Battery health warning. Check/replace battery.
✗ PD(X/96)_TH-Er Incomplete data arrival PLUS a temperature sensor failure. Likely a wiring issue affecting both signal and I2C.
✗ RTC-Er Hardware clock failure. Timestamps will be incorrect. Replace internal button cell battery.
✗ OFFLINE No health report received for > 24.5 hours. Station is completely dead or has lost cellular connection.

💡 Note: OFFLINE Detection

If a station goes completely silent and fails to send a health report for > 24.5 hours, the server will override its historical sensor status and aggressively flag the station as OFFLINE. The Last Seen column provides the exact time since the last successful check-in.

🗺️ How to Navigate This System

📡 DASHBOARD

Your main daily-use page. Shows one row per station (the latest check-in only). Gives you a quick fleet-wide health snapshot. Click any station row to see its full history.

📊 FLEET SUMMARY

Group-level view. Shows each firmware group (KSNDMC-TWS, BIHAR-TRG, etc.) as a card with health stats, firmware conversion progress, and all stations in that group.

🚀 OTA MANAGER

Deploy new firmware to a group. Set the target version, upload the .bin file, and click DEPLOY. All stations in that group will receive the update on their next check-in.

🟣 STATION DETAIL PAGE

Click any station row in Dashboard or Fleet Summary to see its full history (last 100 reports), send commands, upload a custom firmware only for that station, and download its history as CSV.

🟣 Station Detail Column Definitions

Column What It Means
Sig Cellular Signal Strength (dBm). Yellow if worse than -75 dBm.
Net / SPIFFS Net: Number of data records successfully delivered to the server for that day.
SPIFFS: Remaining local storage capacity on the device in kilobytes (kB).

⚡ Command Buttons Explained

Commands are queued on the server and delivered to the device on its next health check-in. The device checks in every 15 minutes, at 11:00 AM daily, or immediately when a sensor fault is detected (one-time report). Commands are never pushed — they wait to be pulled.

↺ REBOOT
Remote Reboot

Queues a REBOOT command. On next check-in, the device receives this, powers off the GPRS modem (GPIO 26 → LOW), then calls ESP.restart(). Use this when a station appears stuck or is showing repeated errors.

↑ BACKLOG
Force Backlog Sync (FTP_BACKLOG)

Immediately triggers upload of the unsent.txt file. Use this if a station was offline and you want to recover missing 15-min data records now.

↑ DAILY
Force Daily File (FTP_DAILY)

Prompts for a date (YYYYMMDD) and triggers upload of that specific daily log file (e.g. dailyftp_20260228.txt). Ideal for recovering the burst data for a specific past day.

🚀 Individual OTA
Deploy Custom Firmware to ONE Station

Available on the Station Detail page. Upload a .bin file targeted at that specific station only. The OTA command is queued immediately. On next check-in, only this station will update — all other stations in the group are unaffected. Use for testing a new firmware on one device before rolling out to a whole group.

GPS FIX
Force GPS Re-acquisition (GET_GPS)

Forces the station to turn on its GPS module, acquire a fresh location fix, save it to persistent memory (SPIFFS), and report the new coordinates on the next check-in. The server also sends this automatically if it detects a missing or invalid GPS location.

CLR FTP
Clear FTP Backlog (CLEAR_FTP_QUEUE)

Safely commands the station to delete its internal ftpunsent.txt queue file. Use this if a station has a massive unrecoverable backlog that is causing HTTP memory/timeout crashes. The server also sends this automatically if the unsent queue exceeds 50 records.

🗑 DELETE
Delete Station Records

Permanently deletes ALL historical records for that station from the database. The station will appear again on next check-in. Use this to clean up test stations or decommissioned units.

🚀 How OTA Update Works (Step by Step)

1

Go to OTA Manager. Find the firmware group you want to update (e.g. SPATIKA-ADDON).

2

Enter the new version number (e.g. 5.80), choose the compiled .bin file, and click DEPLOY.

3

The server now has target = 5.80 for this group. The .bin is stored in /app/builds/FW_S1_SPATIKA_GEN.bin

4

When a station checks in (every 15 min), the server compares its version to the target. If different → response includes "cmd":"OTA_CHECK","cmd_param":"FW_S1_SPATIKA_GEN.bin"

5

The ESP32 downloads the file, flashes it, and reboots. The next check-in will show the new firmware version and the OTA↑ badge will disappear.

💡

Rollback: Set the target version to an older number (e.g. from 5.80 back to 5.78) and upload the older .bin. Same process, works in reverse.

❌ Clearing the Command Queue

Clear OTA Queue

Only removes pending OTA update commands. Use this if you want to cancel a firmware update but keep other queued commands like REBOOT or BACKLOG sync.

Clear ALL Queue

Wipes every single pending command for the station. Resets the device to a clean slate with zero instructions waiting on its next check-in.

🔧 Field Engineer Quick Guide

Station shows CDM or NDM status
  1. Click station → check last 5 health reports
  2. Look at Signal strength trend
  3. If signal OK → check SD card status
  4. Send FTP_NOW command to force a data push
  5. If no improvement → queue REBOOT
  6. If still failing → field visit required
Station shows Low Battery
  1. Check Solar V — should be >12V in daylight
  2. If Solar V is 0 → panel or cable issue
  3. If Solar V OK but Bat low → charge controller fault
  4. Do NOT reboot — may not wake up
  5. Schedule urgent field visit
Station has OTA↑ badge
  1. OTA will happen automatically on next check-in
  2. No action needed — wait 15 minutes
  3. After update, badge disappears
  4. Check health status after update
Station hasn't been seen for >1 hour
  1. Check Net Count today vs. yesterday
  2. If Net Cnt = 0 today → no data at all
  3. Check Reg Fails — high count = SIM issue
  4. Check carrier in CSV export
  5. If no cellular coverage → field visit
